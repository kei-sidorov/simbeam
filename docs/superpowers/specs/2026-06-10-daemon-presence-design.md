# Дизайн: live presence демонов (online/offline маков на клиенте)

Дата: 2026-06-10
Статус: утверждён, готов к плану

## Задача

Клиент (web/debug, в перспективе — нативный) показывает рядом с каждым
запейренным маком точку **online/offline**, которая обновляется **в реальном
времени**: мак выключили/уснул → точка гаснет сама, включили → загорается.

В этом репозитории делаем: presence-канал на брокере (клиент-агностичный
контракт) + точку-индикатор в `web/debug` как референс-реализацию. Нативный
клиент (отдельный репозиторий) потом потребляет тот же канал.

## Контекст (почему так, а не иначе)

- Демон **не открывает портов**: он сам дозванивается до брокера по WS и держит
  соединение постоянно открытым (`ServeSignal` → `serveOnce`, авто-реконнект с
  бэкоффом). Брокер уже трекает это в `b.daemons[daemonID] *daemonConn`
  (`internal/signalbroker/broker.go`). **«Демон онлайн» ≡ «его WS к брокеру
  сейчас жив».** Источник правды о presence уже существует — мы лишь выставляем
  его наружу.
- Брокер **не знает**, с какими маками клиент спарен: пиннинг — на стороне
  endpoint'ов, брокер недоверенный. Список спейренных маков живёт в браузере
  (`localStorage simbeam_macs`: `{signal, daemon, daemonPub, name}`). Поэтому
  presence — это **подписка**: клиент сам присылает список `daemonID`, за
  которыми следить. Брокер не может разослать статусы по своей инициативе —
  он не знает, чьи маки спрашивать.
- При `join` (попытке стрима) брокер уже отвечает `"device offline — wake your
  Mac"`, если демона нет в мапе — то есть presence де-факто уже видно, но только
  в момент полной попытки подключения. Мы выносим его в лёгкий отдельный канал.
- Доступ к presence — **по `daemonID`** (это публичный ключ демона, известный
  спейренным клиентам). Кто знает id — видит online/offline. Без отдельной
  аутентификации: брокер всё равно не может проверить связь клиента с конкретным
  маком (он не знает pinned-наборы), а сам факт «мак включён» — низкий риск.
  Совпадает с текущей моделью доверия.

## Механика (подход A: WS-подписка `watch`)

Presence — **отдельный** канал, не связанный с `join`:

- `join` — «стримить с ЭТОГО одного мака» (видео + управление, тяжёлое
  рукопожатие).
- `watch` — «дай статусы ВСЕГО списка и присылай изменения» (лёгкое, без
  рукопожатия).

Поток:

```
клиент → брокер:   watch  {daemons: [id1, id2, id3]}
брокер → клиент:   presence {states: {id1:true, id2:false, id3:false}}   ← снапшот: все статусы сразу
   …id2 включился…
брокер → клиент:   presence {states: {id2:true}}                          ← дельта: только изменившийся
```

Снапшот и дельта — **один формат** (`states: map[daemonID]bool`); снапшот несёт
все запрошенные id, дельта — один ключ. Клиент мёржит в локальное состояние —
единый код обработки.

## Изменения по компонентам

### 1. Wire-протокол — `internal/signal/message.go`

Новые типы сообщений:

```go
TypeWatch    = "watch"    // client → broker (первое сообщение): наблюдать за списком демонов
TypePresence = "presence" // broker → client: снапшот или дельта статуса
```

Новые поля в `Msg`:

```go
Daemons []string        `json:"daemons,omitempty"` // watch: список daemonID для наблюдения
States  map[string]bool `json:"states,omitempty"`  // presence: daemonID → online (снапшот или дельта)
```

Round-trip нового JSON покрывается тестом в `message_test.go`.

### 2. Брокер — `internal/signalbroker/presence.go` (новый файл)

`handleWS` уже диспатчит по типу первого сообщения. Добавляем ветку:

```go
case signal.TypeWatch:
    b.serveWatcher(c, first)
```

Реестр наблюдателей в `Broker`, под **тем же** `b.mu`, что и `daemons`:

```go
watchers map[*watcher]struct{}

type watcher struct {
    c   *conn
    ids map[string]bool // множество наблюдаемых daemonID
}
```

`serveWatcher(c, first)`:
1. Под `b.mu`: регистрирует watcher **и** строит снапшот из текущего
   `b.daemons` (для каждого запрошенного id: `online := b.daemons[id] != nil`).
2. Отдаёт `presence`-снапшот.
3. Читает из сокета до закрытия (детект обрыва); на выходе — дерегистрация под
   `b.mu`.

`notifyPresence(id string, online bool)` — вызывается из `serveDaemon`:
- при регистрации демона (после вставки в `b.daemons`) → `online=true`;
- в `defer`-разрегистрации (когда демон удаляется из `b.daemons`) → `online=false`.

Реализация: под `b.mu` собрать список conn'ов watcher'ов, у которых
`ids[id]` истинно; **после** снятия лока отправить каждому
`presence {states:{id:online}}` с write-deadline (чтобы мёртвый watcher не
подвесил горутину демона).

**Почему один мьютекс на `daemons` + `watchers`:** иначе появляется щель —
watcher прочитал снапшот «offline», демон зарегистрировался и пушнул дельту
**до** попадания watcher'а в реестр → дельта потеряна, точка навсегда серая.
Общий лок закрывает гонку: либо watcher в реестре до `notifyPresence` (получит
дельту), либо демон в `b.daemons` до снапшота (снапшот покажет online).
Двойная нотификация (снапшот online + дельта online) безопасна — клиент мёржит
идемпотентно.

**Без ложного offline при реконнекте:** перерегистрация демона затирает слот;
`defer` старой горутины проверяет `b.daemons[id] == d` и не удаляет (там уже
новый `d`), значит и `notifyPresence(offline)` не шлёт. Существующая защита.

### 3. Keepalive на брокере — в том же `presence.go` / `broker.go`

Ключевой инсайт: **чистый выход демона** (Ctrl-C, `brew services stop`, краш с
закрытием TCP) рвёт WS → брокер ловит ошибку чтения → offline **мгновенно**.
Heartbeat нужен только для **жёсткого сна Mac**, когда TCP подвисает half-open и
брокер ошибочно считает демон живым.

На conn'ах демонов (и watcher'ов):
- read-deadline + pong-handler, продлевающий дедлайн;
- тикер, шлющий ping.

Дефолты: ping каждые 10s, read-deadline 25s → жёсткий сон детектится за ~10–25s.
Параметры — константы (при необходимости вынести в `Config`).

**Демон не меняем:** gorilla/websocket авто-отвечает pong'ом на ping внутри
`ReadJSON`-цикла, который `serveOnce` и так крутит постоянно. Симметричный
keepalive «демон→брокер» — вне скоупа (существующий авто-реконнект демона это
покрывает).

### 4. Web-клиент — `web/debug/index.html`

- Новый presence-WS, **отдельный** от стримингового `sig`. На загрузке страницы
  (`init`, после `renderMacs`) группируем `loadMacs()` по полю `signal` (на
  практике один брокер → один WS), шлём `{type:'watch', daemons:[…ids]}`.
- На входящий `presence` — мёржим `m.states` в локальную `presenceState`
  (`daemon → bool`), перерисовываем.
- `renderMacs()`: перед каждым маком — точка:
  - 🟢 (зелёная) если `presenceState[daemon] === true`;
  - ⚪ (серая) если `=== false`;
  - «?» (неизвестно) пока снапшот не пришёл.
- Авто-реконнект presence-WS с бэкоффом; на обрыве все точки → «?» (статус
  неизвестен), при восстановлении снапшот их обновит.
- При изменении списка маков (после успешного пейринга / `saveMac`) —
  `subscribePresence()` пере-подписывается (закрыть и переоткрыть WS с актуальным
  списком id).

## Edge-кейсы

- Пустой `watch` (у клиента нет маков) — допустим: клиент просто не открывает
  presence-WS; на стороне брокера пустой список → пустой снапшот.
- Неизвестный `daemonID` (никогда не регистрировался) → в снапшоте `false`.
- Несколько вкладок/клиентов наблюдают один демон — каждый watcher
  самостоятельный; нотификация идёт всем.
- Вытеснение клиента в стриме (`join`) к presence-каналу отношения не имеет —
  это разные сокеты.

## Тесты

- `internal/signalbroker` (по образцу `broker_test.go`, httptest + in-process WS):
  1. демон зарегистрирован → watcher получает снапшот `{id:true}`;
  2. демон отключился → watcher получает дельту `{id:false}`;
  3. watcher подписался **до** появления демона → снапшот `{id:false}`, затем при
     регистрации демона — дельта `{id:true}` (проверяет отсутствие щели).
- `internal/signal/message_test.go`: JSON round-trip полей `Daemons` / `States`.
- Точный тайминг heartbeat не тестируем (хрупко); проверяем лишь, что обрыв
  conn'а демона убирает его из presence (покрыто тестом 2).

## Вне скоупа

- HTTP-поллинг `GET /v1/presence` (подход B) — держим в уме как дешёвый one-shot,
  но сейчас не делаем (YAGNI; противоречит «живому» индикатору).
- Симметричный keepalive «демон→брокер».
- Нативный клиент-потребитель (отдельный репозиторий).
- Аутентификация presence (challenge ключа клиента) — брокер всё равно не
  доказывает связь с конкретным маком; не делаем.
