# Phase 4 — Удалённый доступ через интернет (дизайн)

> Статус: дизайн-спека, до плана реализации. Это **Phase 4** (удалёнка + дистрибуция),
> она НЕ опережает Phase 3 (нативный iPad-клиент). Здесь фиксируется только сетевой/
> транспортный слой удалёнки и его монетизация; рендер/жесты клиента — отдельно (Phase 3).

## Цель

Дать пользователю доступ к симулятору на его Mac с iPad **из любой точки интернета**, с
низкой задержкой, без проброса портов и без выставления Mac в интернет. Флоу «как в happy»:
демон показывает QR → клиент сканирует → поднимается защищённое соединение.

Продукт рассчитан на **широкую аудиторию** (не только автор), поэтому облачная часть
(signaling + STUN/TURN + аккаунты) — это управляемый сервис и одновременно **ров open-core**
(решение №9): сервер-демон OSS, клиент и облако — платные.

## Не-цели (в этой фазе не делаем)

- Свой rendezvous-протокол поверх голого UDP — берём штатный WebRTC ICE.
- Своё шифрование медиа — WebRTC даёт DTLS-SRTP E2E (решение №7).
- Реальные устройства, адаптивный битрейт, новые видеокодеки — см. «Сознательно отложено».
- P2P без рандеву / mDNS-only обнаружение в одной сети — рандеву используем всегда (см. ниже).

## Ключевые факты, на которых стоит дизайн

1. **Рандеву неизбежно.** Два пира за NAT не знают адресов друг друга, пока их не сведёт
   всегда-онлайн signaling. Через signaling течёт только рукопожатие (offer/answer + ICE),
   НЕ видео — поэтому он дёшев.
2. **Одна сеть обслуживается автоматически.** Signaling используется и в одной сети (для
   знакомства), но дальше ICE сам собирает host/srflx/relay-кандидатов, гоняет connectivity
   checks и выбирает путь с наивысшим приоритетом. host↔host (прямой LAN) приоритетнее всего →
   в одной сети видео идёт напрямую по локалке, не трогая STUN/TURN. Ручной логики «мы рядом?»
   не пишем.
3. **Цена сосредоточена в TURN.** STUN stateless и почти бесплатен; signaling брокерит только
   handshake. Дорог только TURN — через relay течёт весь видеопоток. Значит платный гейт
   вешаем именно на TURN, и он совпадает с реальной себестоимостью 1:1.
4. **Демон делает только исходящее.** Он дозванивается до signaling (WSS out) и поднимает
   WebRTC. На Mac не открывается ни одного входящего порта — ноль attack surface, работает
   из-за домашнего NAT без проброса портов.

## Архитектура

```
QR  =  { signalingURL, pairingToken, daemonPubKey }
        │  iPad сканирует
        ▼
[managed signaling]  WSS, всегда онлайн
        │  сводит демон и клиент по pairingToken;
        │  аутентифицирует рукопожатие по daemonPubKey;
        │  отдаёт iceServers (STUN всем; TURN-creds только подписчикам)
        ▼
WebRTC P2P между iPad и Mac:
   • видео  ── H.264 media track ──────────────►  (Phase 2 пайплайн: screenshot→ffmpeg→pion)
   • control + API  ── DataChannel (JSON) ──────►  list/boot/describe + tap/swipe/home/key
   ICE сам выбирает путь: host (LAN) → srflx (STUN) → relay (TURN, платно)
        ▲
Mac: simcastd (Go)  — только исходящие соединения, ноль открытых портов
```

### Компоненты

**1. Signaling-сервис (новый, облачный).** Тонкий WSS-брокер.
- Держит «комнаты» по `pairingToken`. Демон регистрирует комнату; клиент входит по токену из QR.
- Релеит offer/answer + ICE-кандидаты между двумя пирами. Видео через него не идёт.
- Выдаёт клиенту `iceServers`-конфиг (см. гейтинг TURN ниже).
- Zero-knowledge по медиа: даже видя сигналинг, расшифровать поток не может (DTLS-SRTP E2E).
- Дёшев и горизонтально масштабируется (stateless по сессии + общий стор комнат).

**2. STUN/TURN (coturn).**
- STUN — открыт всем (дёшево).
- TURN — только подписчикам, по короткоживущим HMAC-кредам (TURN REST / long-term credential
  mechanism): `username = "<expiry>:<userId>"`, `password = base64(HMAC-SHA1(secret, username))`.
  Креды живут минуты, выписываются auth-сервером при старте сессии.

**3. Pairing / QR.**
- QR несёт `signalingURL`, одноразовый `pairingToken` (короткоживущий) и `daemonPubKey`.
- Демон при старте удалёнки генерит сессионную ключевую пару, регистрируется на signaling под
  `pairingToken`, рисует QR.
- Клиент сканирует → коннектится к `signalingURL`, входит в комнату `pairingToken`.
- `daemonPubKey` аутентифицирует сторону демона на этапе сигналинга (подпись сообщений
  рукопожатия) → защита от MITM на рандеву. Это и есть «защищаем только signaling» из
  ARCHITECTURE; сами медиа уже E2E через DTLS-SRTP.

**4. Транспортные изменения в `simcastd` (Go).**
- Сейчас (Phase 1–2): REST `/api/*` + WS `/session` + WS `/rtc` на localhost.
- Для удалёнки: **весь control и API уезжают в WebRTC DataChannel.** Демон НЕ слушает публичных
  портов; список/бут/`describe`/тачи — это JSON-команды в DataChannel (расширяем формат `/rtc`,
  переиспользуя `parseControl`/`ScaleTap`/`hid` — граница из решения №30).
- Сетевой клиент демона: исходящий WSS к signaling, регистрация комнаты, обмен SDP/ICE.
- Локальный режим (`--web`, localhost) сохраняется как есть для разработки/дебага.

**5. Клиент (iPad, Phase 3 + это).** Сканер QR, WSS к signaling, WebRTC-приём (медиа + DataChannel),
обработка `iceConnectionState`/`connectionState` для апселла (ниже).

## Free vs Paid — гейтинг TURN

При установке сессии auth-сервер решает, что положить в `iceServers` клиенту:

| План | iceServers | Поведение |
|------|-----------|-----------|
| **Подписчик** | STUN + TURN (ephemeral HMAC creds) | коннект гарантирован откуда угодно (relay-fallback) |
| **Free** | только STUN (+ host) | одна сеть → всегда работает; интернет с дружелюбным NAT (~80–90%) → работает; злой симметричный NAT → P2P не пробьётся |

Signaling и STUN открыты всем (копейки, можно rate-limit). Платный — **только TURN**, ровно
дорогой по трафику ресурс.

## Обработка ошибок / деградация

- Free-юзер, P2P не пробился: клиент ловит `iceConnectionState === "failed"` (или `connectionState
  === "failed"`) и показывает апселл: *«Бесплатно relay-инфраструктура не поддерживается —
  будьте в одной сети с Mac или оформите подписку.»* Сообщение технически честное: в одной
  сети TURN физически не нужен.
- Истёкший/использованный `pairingToken`: signaling отклоняет вход в комнату → клиент просит
  пересканировать QR.
- Провал аутентификации по `daemonPubKey`: рвём рукопожатие (возможный MITM).
- Обрыв P2P: как сейчас в Phase 2 — таймаут pion (`OnConnectionStateChange` → закрытие сессии,
  убийство сайдкара + ffmpeg, решение №39).

## Тестирование

- Unit: выписка/проверка TURN-кредов (HMAC), генерация/парсинг QR-payload, подпись/проверка
  сигналинга по `daemonPubKey`.
- Интеграция signaling: две стороны находят друг друга по `pairingToken`, обмен offer/answer/ICE.
- Сетевые сценарии (по возможности эмуляцией NAT): (a) одна сеть → host-путь, TURN не задействован;
  (b) разные сети, дружелюбный NAT → srflx-путь; (c) симметричный NAT, free → `failed` + апселл;
  (d) симметричный NAT, подписчик → relay-путь, видео идёт.
- E2E: QR → скан → видео + control по DataChannel через интернет.

## Открытые вопросы (решаем в плане реализации)

- Аккаунты/биллинг: свой бэкенд или внешний провайдер; где живёт «подписчик?»-проверка.
- Хостинг signaling/coturn: какой провайдер, регионы (латентность TURN важна).
- Формат QR-payload и схема подписи рукопожатия (конкретный crypto-примитив).
- Нужен ли demo/local signaling в OSS-демоне (чтобы self-host был возможен без облака).

## Сознательно отложено (не в этой фазе)

Адаптивный битрейт, ScreenCaptureKit-пайплайн, бандлинг companion'а, нотаризация, реальные
устройства, VP9/AV1 для узкого канала — см. `ARCHITECTURE.md`.
