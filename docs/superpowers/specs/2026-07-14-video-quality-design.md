# Клиентская настройка качества видеопотока

Клиент задаёт качество потока — на аттаче и на лету, не выходя из сессии. Пресетов
на демоне нет: провод несёт числа, пресеты живут в UI iPad-клиента (другой репозиторий).

## Что настраивается

**`scale`** — множитель разрешения относительно источника, `0.25…1.0`.
Множитель, а не абсолютная ширина: клиенту не нужно знать физический размер
устройства, «половина от того, что есть» осмысленна и для SE, и для iPad Pro.

**`bitrate`** — бит/с, `500_000…16_000_000`.

**fps не настраивается.** Один `Screenshot` RPC у idb стоит 71.5–73.9 мс при поллинге
раз в 66.67 мс (`docs/research/2026-06-08-latency-pipeline.md`) — захват уже упёрт в
потолок на 15 fps. Ручка обещала бы то, чего пайплайн не умеет.

Адаптивный битрейт — вне скоупа (`ARCHITECTURE.md`, «Сознательно отложено»).
Здесь только ручной выбор.

## Дефолты и клампы

| | диапазон | sim | browser |
|---|---|---|---|
| `scale` | 0.25 … 1.0 | 0.5 | 1.0 |
| `bitrate` | 500 kbps … 16 Mbps | 8M | 8M |

Дефолты равны сегодняшнему захардкоженному поведению: sim халвит
(`-vf scale=iw/2:ih/2`), browser снимает 390×844 при DPR 1 и кодирует 1:1.
Ноль/отсутствие поля → дефолт бэкенда. Значение вне диапазона → кламп, не ошибка.

## Провод

**`quality` — по `bulk`** (reliable, ordered), не по `control`:

```json
→ {"type":"quality","scale":0.5,"bitrate":8000000}
← {"type":"quality","scale":0.5,"bitrate":8000000}   // применённое, после клампа
← {"type":"error","msg":"no simulator attached"}
```

`control` имеет `maxRetransmits: 0` — на плохой сети сообщение теряется. Плохая сеть —
ровно тот случай, ради которого фича существует, поэтому ползунок едет по надёжному
каналу. Ответ обязателен: контракт `bulk` требует всегда отвечать (решение №87).
Применённые значения в ответе достаются даром и показывают клампинг.

**`attach` — по `control`**, получает те же поля опционально (стартовые):

```json
{"type":"attach","udid":"…","scale":0.5,"bitrate":8000000}
```

`attached` не меняется — `w`/`h` остаются нативными. Эха применённых значений там нет:
единственный клиент в репо (`web/debug/index.html:492`) вообще не читает `w`/`h`,
поле уже мёртвое. Реальное разрешение трека клиент узнаёт из `<video>.videoWidth`.

## Смена на лету

`quality` → кламп → сохранить в сессии → внутренний re-attach текущего udid.
`doAttach` уже начинается с `stopAttachment()`, так что это существующий путь с новым
аргументом, а не новая машина состояний. Re-attach уходит в горутину: `handleBulk`
блокировать нельзя, там же сидит `doScreenshot` со своим 15-секундным таймаутом.

Трек не пересогласовывается. Кадр замирает на последнем до нового IDR, который ffmpeg
отдаёт на старте; смена разрешения — новые SPS/PPS + IDR, декодер ресинкается
(короткий GOP, решение №50).

`quality` без активного аттача → текстовая ошибка. Стартовые значения — дело `attach`.

## Совместимость

**Старый клиент → новый демон:** работает на дефолтах. `attach` без полей → нули →
кламп подставляет дефолт бэкенда = сегодняшнее поведение. `quality` не шлётся.
Канала `bulk` может не быть вовсе (дебаг-клиент открывает только `control`) — роутинг
по метке в `OnDataChannel` это переживает.

**Новый клиент → старый демон** — асимметрия, которая укусит: демон обновляется через
`brew` вручную, а iPad-клиент прилетает из App Store сам.

- `quality` в `bulk` → старый демон ответит `{"type":"error","msg":"unknown bulk type \"quality\""}`.
  Это **единственный надёжный способ детекта** — по нему клиент прячет ползунок.
- `attach` со `scale`/`bitrate` → старый демон **молча проигнорирует** незнакомые поля
  (дефолт `encoding/json`) и приаттачит на своих числах. Клиент будет думать, что
  применил. Щупать версию нужно `quality`, а не `attach`.

## Реализация

- `internal/encoder/ffmpeg.go` — `ffmpegArgs(fps, scale, bitrate)`; `Encode`/`EncodeFullRes`
  схлопываются в одну функцию с параметром (`scale == 1.0` → `-vf` не добавляется).
  Фильтр — `scale=trunc(iw*S/2)*2:-2`, и явная чётность здесь обязательна:
  scale-фильтр сам её не держит, наивный `scale=iw*S:ih*S` при нечётном результате
  молча чинится в `h264_videotoolbox`, но `libx264` отказывается стартовать
  (`width not divisible by 2`). Захардкоженный 0.5 этого не задевал; любая
  клиентская ручка — задела бы, и уронила бы именно Linux-демо. Проверено на обоих
  энкодерах.
- `internal/server/backend.go` — `QualityOpts{Scale, Bitrate}` + `Resolve(defScale)`;
  `Backend.Attach(ctx, udid, opts)`.
- `internal/backend/{sim,browser}` — принимают opts, свои дефолты `scale`.
  В browser не путать с `Options.Scale` — то deviceScaleFactor источника, это кодек.
- `internal/server/control.go` — поля в `controlMsg`.
- `internal/server/attach.go` — `doAttach(udid, opts)`, сессия помнит текущие.
- `internal/server/bulk.go` — тип `quality`, `doQuality`.

Тесты: табличный на `ffmpegArgs` (scale/bitrate/чётность), юниты `Resolve`,
bulk-диспатч на `quality` без аттача и с ним.
