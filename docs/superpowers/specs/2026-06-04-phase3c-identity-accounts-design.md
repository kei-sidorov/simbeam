# Phase 3C — Постоянная парность, аккаунты-по-ключам, подписки (дизайн)

> Статус: дизайн-спека, до плана реализации. Следующий шаг — `superpowers:writing-plans`.
> Контекст: продолжение Phase 3 (удалёнка). 3a — control-plane/video-on-demand (сделано),
> 3b — signaling/рандеву (сделано, решения #51–54). 3C добавляет **постоянную парную связь**,
> чтобы клиент и Mac находили друг друга и переподключались без повторного QR, плюс минимальный
> слой подписок для гейтинга TURN. За 3C идёт Phase 4 (дистрибуция + прод-облако + чек Apple).

## Цель

Сейчас (после 3b) пейринг одноразовый: QR/URL → один сеанс. Если у клиента (браузер/iPad)
пропала сеть и вернулась, или человек ушёл из дома и хочет подключиться позже — нужно заново
сканировать QR. 3C даёт: **спарились один раз ключами → дальше находим друг друга по сохранённой
идентичности и переподключаемся автоматически, без QR.** Плюс тонкий слой подписок (одна таблица),
чтобы гейтить TURN по реальной подписке вместо стаба `--grant-turn`, с возможностью смотреть данные
в БД глазами.

Главная идея — развести **знакомство** (один раз, QR/URL, обмен постоянными ключами) и
**переподключение** (сколько угодно, автоматически, по ключам).

## Не-цели (в этой фазе не делаем)

- **Биллинг / проверка чека Apple.** Подписка пока *client-asserted*: клиент сообщает серверу
  «подписка до даты X», доверяем Apple на не-jailbreak устройствах. Усиление (валидация
  StoreKit-чека, App Store Server Notifications) — Phase 4, встанет в тот же endpoint, флипнув
  поле `source`, без смены схемы.
- **Эмуляция протухания / фейк-часы.** Не нужна — ставим `expires_at` вперёд/назад руками.
- **Sign in with Apple** и серверный OAuth/JWT-слой — берём аккаунты-по-ключам (Ed25519, уже есть).
- **Таблицы `accounts`/`devices` и серверное восстановление списка устройств** — список Mac'ов и
  ключи живут на концах (localStorage / Keychain c iCloud-синком); восстановление — через iCloud
  Keychain.
- **Нативный iOS-клиент** — отдельный репозиторий, позже. Валидация — в браузере.
- **Postgres** — сейчас SQLite за интерфейсом `Store`; миграция позже сменой драйвера.
- **Горизонтальный брокер / Redis, App Store Server Notifications, wake сонного Mac, trickle ICE,
  TLS/домен, прод-хардненинг брокера** — Phase 4 / deploy.

## Ключевые решения (резюме, развёрнуто ниже)

1. **Авторизация связи — концы по ключам (peer-pinning), брокер untrusted.** Клиент пиннит pubkey
   конкретного Mac, демон пиннит pubkey клиента; при коннекте стороны проверяют друг друга сами,
   брокер только релеит по `daemonID` и **не решает** доступ.
2. **БД минимальна — одна таблица `subscriptions`.** Ключи и список Mac'ов — на концах. Серверного
   восстановления нет (iCloud Keychain).
3. **Постоянные идентичности.** Демон — долгоживущий ключ на диске (`daemonID` = его pubkey) +
   набор запиненных клиентских ключей. Клиент — своя пара + локальный список запиненных Mac'ов.
   Один клиентский ключ = идентичность для пиннинга **и** для подписки.
4. **Постоянное присутствие демона.** Демон держит постоянный исходящий WSS к брокеру
   (auto-reconnect), всё время зарегистрирован под `daemonID`. Presence у брокера — в памяти.
5. **Первый пейринг — явное окно на демоне** с одноразовым секретом `S` (HMAC-доказательство, `S`
   брокеру не виден); дальше — только взаимный key-challenge.
6. **Подписка — отдельный аутентифицированный HTTPS-endpoint** с двумя подписями (app-secret HMAC
   слабый + account Ed25519 настоящий), idempotent last-write-wins; гейт TURN читает `Store`.
7. **Один серверный процесс** `simbeam-signal`: брокер WS + HTTP-API + SQLite (`Store`-интерфейс).

## 1. Идентичности и модель связи

### Ключи (постоянные, на концах)

- **Демон** при первом старте генерит долгоживущую пару Ed25519, кладёт на диск
  (`~/.simbeam/identity.key`, права 0600). Публичная часть base64 = **`daemonID`** — одновременно
  стабильный адрес на брокере и криптоудостоверение. Рядом — **набор запиненных клиентских pubkey**
  (`~/.simbeam/clients.json`): кому разрешено подключаться (+ опц. имя устройства).
- **Клиент** (браузер/iPad) генерит свою пару Ed25519. Приватный ключ — в `localStorage` (браузер)
  / Keychain с iCloud-синком (iPad). Хранит локально **список запиненных Mac'ов**:
  `[{daemonID, daemonPubKey, name}]`. **Один клиентский ключ** играет две роли: идентичность для
  пиннинга у демона **и** идентичность подписки (ключ = `client_pubkey` в таблице).

### Присутствие

Демон держит **постоянный** исходящий WSS к брокеру (auto-reconnect + экспоненциальный backoff),
всё время зарегистрирован под `daemonID`. Брокер хранит presence **в памяти** (`map[daemonID]→conn`),
ничего durable. Mac уснул/выключен → WSS рвётся → регистрация снята → клиенту отдаём «устройство
офлайн, разбуди Mac».

### Подключение (и переподключение — один путь)

1. Клиент берёт `daemonID` из локального списка, входит на брокер «хочу к `daemonID`».
2. Брокер сводит с демоном (если онлайн) и релеит **взаимный challenge-response**: демон подписывает
   nonce постоянным ключом → клиент проверяет по запиненному `daemonPubKey`; клиент подписывает
   nonce → демон проверяет, что `clientPubKey` есть в его pinned-set. **Сервер тут ничего не решает.**
3. Прошли — обычное WebRTC-рукопожатие (offer/answer, как в 3b: браузер-offerer, демон-answerer,
   non-trickle; answer подписан постоянным ключом демона). Дальше P2P как сейчас (видео-трек +
   control-DataChannel, attach видео по `attach <udid>`).
4. Обрыв сети → клиент молча повторяет с шага 1. **QR не трогаем.** Disconnect ловит pion
   (`OnConnectionStateChange`, #39) на стороне демона; клиент — `connectionState`.

### Что меняется относительно 3b

Механика 3b сохраняется (брокер-релей, offer/answer, control-DataChannel, attach видео). Меняется
только: ключ демона стал **постоянным** (был сессионный), одноразовый `pairingToken` стал
**стабильным `daemonID`**, перед WebRTC-рукопожатием добавлен **взаимный challenge-response**, и
демон **висит постоянно** (был one-shot, #51 в этой части пересматривается).

## 2. Первый пейринг (enrollment)

Пейринг — **явное действие на демоне** (не фоновое). Пользователь говорит «спарить новое
устройство». Демон генерит **одноразовый секрет `S`** (короткий TTL, single-use) и показывает
pairing-payload: `daemonID (= daemonPubKey)` + `signalURL` + `S`. В браузере — URL-фрагмент
(`#signal=…&daemon=<pubkey>&pair=<S>`); на iPad позже — QR. Вне этого окна демон новые enrollment'ы
не принимает (анти-абьюз).

### Две роли в payload (как в 3b #45/#54, но теперь постоянные)

- `daemonPubKey` из payload → клиент **пиннит** его. **Анти-MITM:** брокер не подменит ключ демона.
- `S` → доказывает демону, что клиент **авторизован** добавиться. **Анти-«чужой»:** кто-то,
  узнавший `daemonID`, без `S` не впишется.

### Флоу (через тот же брокер; `S` брокеру не виден)

1. Клиент (сгенерил пару, если ещё нет) входит на брокер к `daemonID`, шлёт `clientPubKey` +
   `HMAC(S, clientPubKey‖nonce)`.
2. Демон проверяет HMAC своим `S` (клиент знает `S`), `S` не истёк/не использован → **пиннит
   `clientPubKey`** в набор, помечает `S` использованным.
3. Демон подписывает nonce постоянным ключом → клиент проверяет по `daemonPubKey` из payload (демон
   правда владеет приватным ключом, не реплей чужого pubkey).
4. Готово: клиент сохранил `{daemonID, daemonPubKey, name}`, демон — `clientPubKey`. Имена
   (hostname Mac / метка устройства) — опционально, для UI.

### Единый путь коннекта

«Первый раз» = коннект с доказательством `S` + взаимная подпись (выше). «Потом» = тот же коннект,
но **только** key-challenge из §1 (демон видит `clientPubKey` уже в наборе → `S` не требует).

### Ревокация

«Отвязать устройство» на демоне = убрать `clientPubKey` из набора (локально, без сервера). Тот
клиент больше не подключится. Потерял ключ без iCloud → пере-пейринг новым `S`.

## 3. Подписки

### Хранилище — одна таблица в SQLite за интерфейсом `Store`

```
subscriptions(
  client_pubkey  TEXT PRIMARY KEY,  -- тот же ключ, что в пиннинге (§1)
  product_id     TEXT,
  expires_at     TEXT,              -- RFC3339; дата от StoreKit (пока client-asserted)
  issued_at      TEXT,              -- RFC3339; время отчёта клиента (для упорядочивания)
  source         TEXT,              -- 'client' сейчас; 'apple-verified' потом
  updated_at     TEXT)              -- RFC3339; серверные часы записи
```

Подписка привязана к `client_pubkey`. iPad и браузер — разные ключи → разные подписки; платящий
клиент в проде = iPad-app, браузер — dev/free.

`Store` — тонкий Go-интерфейс (методы под наши операции), реализация над `database/sql`. Портируемый
SQL (`INSERT … ON CONFLICT … DO UPDATE` есть и в SQLite, и в Postgres), `now` передаётся параметром
(не `CURRENT_TIMESTAMP`), даты — текст RFC3339. Миграция на Postgres = вторая реализация интерфейса +
смена драйвера/DSN.

### Транспорт — аутентифицированный HTTPS на том же сервере

```
POST /v1/subscription
  body: { client_pubkey, product_id, expires_at, issued_at }
  X-App-Sig:     base64(HMAC-SHA256(APP_SECRET, canonical))   // «наш билд» — слабый барьер
  X-Account-Sig: base64(Ed25519(client_key, canonical))       // «это тот аккаунт» — настоящая auth
  canonical = client_pubkey ‖ product_id ‖ expires_at ‖ issued_at   (фиксированный разделитель)
```

- `APP_SECRET` — общий секрет, в приложении (приватный репо) + env сервера (`SIMCAST_APP_SECRET`).
  **Честно: это обфускация, не криптограница** — вынимается реверсом из бинаря (приватность репо не
  защищает). Настоящая граница «не платил — не продлил» = серверная проверка чека Apple, **отложена**;
  встанет в *тот же* endpoint, флипнув `source: 'client' → 'apple-verified'`, без смены схемы.
- `client_key` подпись — настоящая привязка: только владелец приватного ключа продлит свою подписку.
- Клиент шлёт **на покупку, на foreground, на background**. Сервер: проверил **обе** подписи →
  если `issued_at` новее сохранённого → апсерт (last-write-wins по `issued_at`); иначе/без изменений
  → no-op. В обоих случаях `200`. Идемпотентно, спам с foreground/background безопасен.
- Время для сравнения — **серверные часы**, не клиентские; `issued_at` только для упорядочивания.
  Слабая защита от replay: отвергать слишком старый `issued_at` (щедрое окно из-за расхождения
  часов; при статичном `APP_SECRET` защита всё равно слабая — гигиена).

### Гейт TURN (важная связка)

При коннекте брокер **и так** аутентифицирует `clientPubKey` через взаимный challenge-response (§1).
Поэтому подписку он берёт «бесплатно»: один read из `Store` по уже **проверенному** `client_pubkey`,
сравнивает `expires_at > серверные_часы` → выдаёт `iceServers` с TURN (ephemeral HMAC creds, #54) или
STUN-only. Это **на рукопожатии, не в горячем медиапути**. Заменяет стаб `--grant-turn`.

**Не путать две решалки:** доступ к *подключению* решают концы по ключам (брокер untrusted, §1);
доступ к *TURN-реле* решает брокер по подписке — это его законная работа (он выдаёт `iceServers`).
Разные решения, не противоречат.

## 4. Серверный процесс

**Один бинарь `simbeam-signal`** (расширяем существующий) — три функции в одном процессе:

1. **Брокер WS (`/ws`)** — как в 3b, плюс: постоянная регистрация демона по `daemonID`, релей
   взаимного challenge-response (§1), гейт TURN по подписке (§3).
2. **HTTP-API (`/v1/subscription`)** — управление подпиской (§3). Опционально аутентифицированный
   `GET /v1/subscription/me` — чтобы браузер-стенд показывал текущее состояние (основной способ
   смотреть — сам файл SQLite).
3. **SQLite за `Store`** — файл принадлежит этому процессу.

**Почему один процесс:** SQLite — локальный файл; один процесс снимает кросс-процессные блокировки;
проще для MVP. Разделить на сервисы — потом, интерфейс `Store` позволит.

Флаги: `--addr`, `--db <path>`, `--stun`, `--turn`, `--turn-secret`, `--turn-ttl`; `APP_SECRET` — из
env. Стаб `--grant-turn` **убираем** (гейт читает `Store`; оба тарифа в тесте даём эмуляцией
подписки).

**Демон `simbeamd`** получает:
- постоянную идентичность (ключ на диске) + pinned-clients store + команду ревокации;
- режим **persistent serve** (`serve --signal …`) — коннектится навсегда с auto-reconnect под
  `daemonID`, обслуживает реконнекты запиненных клиентов;
- **pairing-окно** (`--pair` или явный локальный триггер) — открывает одноразовый enrollment,
  печатает pairing-URL c `S`, после привязки продолжает serve.

Точная форма pairing-триггера (флаг старта vs локальный сигнал/endpoint работающему демону) — деталь
плана.

## 5. Тест-стенд в браузере (`web/debug`)

Браузер играет роль будущего iPad по всему флоу — **до единой строчки Swift**:

- **Идентичность:** генерит/показывает `client_pubkey` (= аккаунт), кнопка «сбросить» (новый аккаунт
  для тестов). Приватный ключ — в `localStorage` (WebCrypto Ed25519).
- **Пейринг:** открыл pairing-URL (фрагмент) → enrollment → запинил Mac, он в списке «мои Mac'ы».
- **Мои Mac'ы:** сохранённые `{daemonID, daemonPubKey, name}`; клик → коннект (тот же reconnect-путь);
  виден статус соединения.
- **Панель эмуляции подписки:** `product_id` + date-picker `expires_at` + «Применить» → шлёт
  **настоящий** подписанный `POST /v1/subscription` (dev-`APP_SECRET` + подпись клиентским ключом).
  Замена StoreKit. Протухание по времени не эмулируем — ставим дату вперёд/назад.
- **Индикатор TURN:** после коннекта показывает «iceServers содержат TURN: да/нет» — глазами видно,
  что гейт сработал по подписке.
- **Reconnect:** перезагрузка вкладки / обрыв сети → авто-переподключение по сохранённому `daemonID`,
  без QR.

**Инспекция БД:** открываешь файл SQLite в DB Browser / `sqlite3` → видишь строку `subscriptions`.

**Secure context:** `http://localhost` браузеры считают secure, поэтому WebCrypto Ed25519 и весь
флоу **работают локально по `ws://localhost` без TLS** — тестирование не блокируется доменом/сертом.

## 6. Тестирование

- **Unit (TDD, герметично, Go):**
  - генерация/персист/загрузка ключа демона (права файла);
  - pinned-clients store: add / remove / contains;
  - проверка HMAC-доказательства `S` + one-time/TTL `S`;
  - взаимный challenge-response (sign/verify nonce в обе стороны);
  - endpoint подписки: проверка обеих подписей (app-secret HMAC + account Ed25519), idempotent
    upsert, игнор не-по-порядку по `issued_at`, логика `active = expires_at > now`;
  - `Store` над SQLite: upsert, read, active-subscription (с `now` параметром → портируемо);
  - брокер: persistent register, маршрутизация реконнекта по `daemonID`, peerLeft, гейт читает `Store`.
- **Интеграция (герметично, как `TestRemotePairingEndToEnd` из 3b):** pion-«браузер» + реальный
  брокер + временный SQLite + stub Companion (без idb / браузера / сети):
  - первый пейринг с `S` → взаимный пиннинг;
  - реконнект по `daemonID` без `S` → key-only challenge → коннект;
  - подписанный `POST /v1/subscription` → сохранилось → TURN дали; без/протухшая подписка → STUN-only.
- **Ручная валидация в браузере (стенд, localhost):** весь флоу + негатив (битый `daemonPubKey` →
  MITM-фейл; неверный/отсутствующий `S` → enrollment отклонён); инспекция БД через файл SQLite.
- **Deploy-only (не локально, помечаем):** реальный NAT/TURN-реле под подпиской, TLS/домен, настоящий
  чек Apple — Phase 4.

## Затрагиваемые компоненты (ориентир, детали — в плане)

- `internal/signal` — добавить: persist/load ключа, pinned-store, HMAC-доказательство `S`, взаимный
  challenge-response, канонизацию payload подписки. (Существующие Ed25519/HMAC/TURN-примитивы
  переиспользуем.)
- `internal/signalbroker` — persistent-presence по `daemonID`, релей challenge-response, гейт TURN
  через `Store`; HTTP-роут `/v1/subscription`.
- `internal/store` (новый) — интерфейс `Store` + реализация над SQLite (`database/sql`).
- `internal/server` (демон) — постоянная идентичность, pinned-store, persistent serve + reconnect,
  pairing-окно.
- `cmd/simbeam-signal` — флаги `--db`, env `APP_SECRET`; убрать `--grant-turn`.
- `cmd/simbeamd` — persistent serve, pairing-триггер.
- `web/debug/index.html` — идентичность, пейринг, «мои Mac'ы», панель подписки, индикатор TURN,
  авто-reconnect.

## Граница фаз

3C даёт постоянную парность + аккаунты-по-ключам + подписки + гейт. **Phase 4:** Homebrew-дистрибуция,
задеплоенный managed-сервер (TLS/домен), серверная проверка чека Apple (флип `source`), позже —
Postgres (через `Store`) и горизонтальный масштаб.

## Решённые в брейншторме развилки

- Авторизация связи: **peer-pinning** (концы по ключам), брокер untrusted.
- Объём БД: **только `subscriptions`**; ключи и список Mac'ов на концах; восстановление — iCloud Keychain.
- Аккаунты: **по ключам** (не Sign in with Apple).
- БД-движок: **SQLite сейчас** за `Store`, миграция на Postgres позже.
- Биллинг: **нет**; подписка client-asserted с двумя подписями; усиление чеком Apple — Phase 4.
