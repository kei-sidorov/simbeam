# Лог решений (ADR-lite)

Короткие записи принятых решений. Новое решение — новая строка. Если решение меняем —
не удаляем старое, а добавляем новое с пометкой «отменяет №…».

| # | Решение | Почему |
|---|---------|--------|
| 1 | Скоуп — только симуляторы, не реальные устройства | резко упрощает; idb силён на симуляторах |
| 2 | Сидим на `idb_companion`, не пишем свой движок видео/ввода | готовое, нативное, работает; не переписываем работающее |
| 3 | Говорим gRPC напрямую с companion, `fb-idb` (python) не используем | убираем питон-прослойку и двойной хоп |
| 4 | Демон на **Go** | после делегирования видео в idb демон = сетевой перекладчик байтов |
| 5 | WebRTC-сервер на **pion** | лучший серверный WebRTC; отдаёт готовый H.264 без энкодинга |
| 6 | Клиент на **Swift + GoogleWebRTC** | приёмная сторона WebRTC на iOS — проторённый путь |
| 7 | MVP без NAT-traversal/TURN; своё E2E НЕ пишем | WebRTC уже даёт DTLS-SRTP; relay нужен только для удалёнки позже |
| 8 | Дистрибуция: GoReleaser + Homebrew tap, prebuilt бинари | без нотаризации (brew не quarantine), `depends_on idb-companion` |
| 9 | Open-core: сервер OSS, клиент платный | ров — в UX клиента и облачной инфре, не в доступе к серверу |
| 10 | Первый прототип — JPEG-over-WS, WebRTC вторым шагом | увидеть результат за вечер, не утонуть в WebRTC сразу |
| 11 | Module path `github.com/kei-sidorov/simcast` | по схеме `github.com/<user>/simcast`, выбрано пользователем |
| 12 | `internal/companion` парсит вывод `--list 1` построчно, беря только строки-JSON с `udid` | companion мешает JSON-строки девайсов с диагностикой на stdout; objc-warning на stderr — парсим устойчиво к мусору |
| 13 | Фильтруем `type != "Simulator"` в `List()` | скоуп — только симуляторы; companion заодно перечисляет подключённые реальные устройства |
| 14 | Версию берём из `--version` (JSON `build_time`/`build_date`), не из brew | единственный источник версии у самого бинаря; brew-версия (1.1.8) — это пакет, не отчёт companion |
| 15 | Phase 1: кадры через `video_stream(format=MJPEG)`, не screenshot-поллинг | companion сам отдаёт поток JPEG — ровно «JPEG over WS», без транскодинга, реальный fps; структурно ближе к Phase 2 |
| 16 | Phase 1: транспорт = REST (list/boot) + 1 WS на сессию (кадры↓ binary, тапы↑ JSON) | разные природы взаимодействий; WS full-duplex закрывает стрим+ввод одним соединением |
| 17 | simcastd спавнит companion-сайдкар по требованию (на WS-коннект), бут — отдельной командой | вертикальный срез одной кнопкой; бут управляется клиентом (список+boot), стрим только к booted |
| 18 | Пакеты: `companion` (CLI lifecycle) отдельно от `idb` (gRPC describe/video/hid) | две природы companion'а (CLI-флаги vs gRPC-сервер) — разные границы |
| 19 | Сгенерённые gRPC-стабы (`internal/idbpb`) коммитим в git; команды через Makefile | сборка у потребителя без protoc; виден diff стабов |
| 20 | Координаты тапа — нормализованные `[0,1]` от клиента, сервер масштабирует по `describe` | resolution-independent, не зависит от scale_factor кадра; px-vs-points решаем живым тапом |
| 21 | Debug-клиент — отдельный файл, раздаётся только при `--web`; бинарь самодостаточен по API | клиент только для дебага, не поставляется; реальные клиенты подключатся к REST+WS |
